\chapter{Introduction}
 
\chapter{Background and Related Work}
This chapter establishes the theoretical and empirical foundations of the thesis. \\
First, we introduce the core concepts underlying benchmark-based evaluation of code LLMs and machine unlearning, and then we survey the releant literature on code generation benchmarks, unlearning techniques, and data leakage in code LLMs.

\section{Code Generation Benchmarks}
A benchmark is a standardized collection of tasks and evaluation criteria. It allows the research community to compare the capabilities of different models under controlled, reproducible conditions. In code generation, a benchmark generally consists of a prompt (a natural-language description or partial code stub), an expected output (a function or program), and one or more test cases acting as an oracle for functional correctness. The evaluation score is computed by running model-generated code against these test cases.\\

As a running example, consider the following problem prompt:

	\centering

\begin{lstlisting}[style=pythonstyle]
	def is_palindrome(s: str) -> bool:
		""" Return True if the given string s is a palindrome,
		i.e., it reads the same forwards and backwards.
		Ignore case differences.
		>>> is_palindrome("racecar")
		True
		"""
\end{lstlisting}
	\end{tcolorbox}
\end{figure}

It produces a possible solution:

	s = s.lower()
	return s == s[::-1]
\end{lstlisting}
\end{tcolorbox}
\end{figure}

The generated code is then executed in a sandboxed environment against a predefined set of test cases. If the solution passes all the following test cases, it is marked as correct:

	\centering

\begin{lstlisting}[style=pythonstyle]
	assert is_palindrome("racecar") == True
	assert is_palindrome("hello") == False
	assert is_palindrome("Madam") == True
	assert is_palindrome("") == True
	assert is_palindrome("A") == True
\end{lstlisting}
\end{tcolorbox}
\end{figure}

The primary metric for measuring functional correctness in code generation benchmarks is $pass@k$, as defined by \citet{DBLP:journals/corr/abs-2107-03374}. Given a problem, the model generates $n \geq k$ independent code samples. The $pass@k$ score is then computed analytically via a closed-form unbiased estimator over all $n$ samples, rather than generating exactly $k$. Generating more samples than strictly required is important to account for the inherent stochasticity of LLMs: a single sample may fail due to sampling variance rather than a genuine lack of capability, so evaluating over a larger pool yields a more reliable estimate. In practice, $pass@1$ computed with low-temperature sampling is the most commonly reported variant, as it directly reflects the probability that a single generation attempt produces a correct solution.\\

This execution-based evaluation pipeline is the standard approach across code generation benchmarks. It provides an objective, binary signal of correctness — either the code passes all test cases or it does not — avoiding the ambiguity of surface-level textual metrics such as BLEU (\citet{BLUEamethodforautomaticevaluationofmachinetranslation}) or CodeBLEU (\citet{ren2020codebleumethodautomaticevaluation}). These metrics were originally designed for natural language and may assign high scores to syntactically similar but functionally incorrect programs. \citet{Evtikhiev_2023} confirm this limitation empirically, showing that BLEU and CodeBLEU correlate poorly with human judgments of code correctness and can rank functionally incorrect outputs above correct ones.\\

Benchmark evaluation also requires careful test suite design. Inadequate or trivially simple tests can yield inflated accuracy figures that do not reflect true model capability (\citet{zhang2024naturalcodebenchexaminingcodingperformance}; \citet{ouyang2025dscodebenchrealisticbenchmarkdata}; \citet{koohestani2025benchmarkingaimodelssoftware}). For instance, \citet{ouyang2025dscodebenchrealisticbenchmarkdata} reports that DS-1000 provides an average of only 2.1 test cases per problem — a limitation that motivates DSCodeBench's design of 200 test cases per problem to ensure broader coverage of code behavior, including edge cases and intermediate states.\\

A further threat to benchmark validity is data contamination: benchmark problems or their solutions may be present in a model's pretraining or fine-tuning corpus, causing evaluation scores to reflect memorization rather than genuine generalization. This is a particularly acute problem in the code domain, as widely used benchmarks such as HumanEval and MBPP are publicly available across numerous repositories and may have been seen by a model prior to evaluation. Contaminated models perform significantly better on problems encountered during training than on genuinely unseen ones (\citet{riddell2024quantifyingcontaminationevaluatingcode}; \citet{matton2024leakagecodegenerationevaluation}; \citet{zhou2025lessleakbenchinvestigationdataleakage}). These findings collectively highlight how common and impactful benchmark contamination is in code LLMs, which directly motivates the focus of this thesis.

\section{Machine Unlearning}
Understanding machine unlearning requires first understanding how LLMs acquire knowledge. Modern LLMs are trained in two main phases (\citet{brown2020languagemodelsfewshotlearners}).  In the first phase, called pre-training, the model is exposed to a huge amount of raw text and code; trillions of tokens drawn from online resources, code repositories, books. Then, it is trained to predict the next token in a sequence. Through this process, the model encodes: world knowledge, language patterns, and coding conventions into its parameters. Any information present in the pre-training corpus may be memorized, including benchmark problems and their solutions. 
\par\vspace{1em}
In the second phase, the pre-trained model is fine-tuned on smaller, curated datasets to specialize it for a particular task or to align it with human preferences. This takes the form of supervised fine-tuning (SFT) on instruction-response pairs, or reinforcement learning from human feedback (RLHF), which uses human ratings to steer outputs toward being more helpful, harmless, and honest (\citet{ouyang2022traininglanguagemodelsfollow}). For code-specific models, fine-tuning typically involves additional training on curated code datasets, commit histories, or synthesized instruction-following examples. This pipeline is now standard for building capable code LLMs such as CodeLlama, DeepSeek-Coder, and Qwen-Coder (\citet{rozière2024codellamaopenfoundation}; \citet{hui2024qwen25codertechnicalreport}). 
\par\vspace{1em}
The knowledge absorbed during the pre-training phase are embedded in the weights of the model and it is not easy to remove them.
Fine-tuning on new data is not a  guarantee that previously memorized information is erased — (\citet{hong2024dissectingfinetuningunlearninglarge}) show through activation patching experiments that fine-tuning-based unlearning methods merely alter the model's knowledge retrieval process rather than genuinely removing the knowledge from its parameters, which remains largely recoverable. This is the problem that machine unlearning addresses.
\par\vspace{1em}
Machine unlearning (MU) refers to the specific removal of some training data or knowledge from a trained model, without retraining from scratch. Originally motivated by data privacy regulations (GDPR's right to be forgotten \citet{7163042}), MU has expanded to address copyright removal, safety alignment, bias mitigation, and hallucination reduction in LLMs (\citet{liu2024rethinkingmachineunlearninglarge}). In the unlearning literature, the data to be removed is often referred to as the "forget set", while the data whose knowledge should be preserved is called the "retain set". The unlearning process must ensure that the model's performance on the retain set remains intact while effectively erasing knowledge from the forget set.
\par\vspace{1em}
Exact unlearning — retraining from scratch on the dataset with the target data removed — is the most reliable approach but is computationally infeasible for large models. Research has therefore concentrated on approximate unlearning methods that modify a trained model post-hoc. These methods must balance two competing objectives: unlearning efficacy, meaning that targeted knowledge is effectively erased, and utility preservation, meaning that general model capabilities outside the unlearning scope remain intact.
\par\vspace{1em}
Gradient Ascent (GA) is one of the most common used unlearning approaches. Rather than minimizing the prediction loss, as in standard training, GA maximizes the loss on the data to be forgotten (\citet{yao2024largelanguagemodelunlearning}). However, during the optimization, GA alone can be sensitive to the choice of hyperparameters, which can lead to unlearning failures as catastrophic collapse  (\citet{liu2024rethinkingmachineunlearninglarge}). This has given rise to several improved variants.
\par\vspace{1em}
Gradient Difference (GradDiff) is the combination of GA on the forget set with gradient descent on the retain set and a KL divergence regularization term to minimize the divergence between the unlearned model and the original model on a reference dataset, preventing catastrophic deterioration of general capability (\citet{hong2024dissectingfinetuningunlearninglarge}).
\par\vspace{1em}
Negative Preference Optimization (NPO) treats the forget set exclusively as negative examples: it turns the unlearning problem into a minimization problem mitigating the catastrophic collapse compared to GA (\citet{liu2024rethinkingmachineunlearninglarge}; \citet{jia2025waglestrategicweightattribution}).
\par\vspace{1em}
Preference Optimization (PO)  takes the opposite direction from NPO: instead of treating forget data as negative examples, it introduces targeted unlearning responses ("I don't know") as positive examples, producing a bounded forget loss (\citet{jia2025waglestrategicweightattribution}).
Another variant transforms GA into a gradient descent approach by minimizing the likelihood of predictions on relabeled forgetting data, replacing the original responses with generic or uninformative alternatives (\citet{yao2024largelanguagemodelunlearning}; \citet{liu2024rethinkingmachineunlearninglarge}).
\par\vspace{1em}
A related approach is Task Arithmetic (\citet{ilharco2023editingmodelstaskarithmetic}; \citet{cai2026perparametertaskarithmeticunlearning}): a task vector is defined as the element-wise difference between a fine-tuned model's weights and the original pre-trained model's weights.
Negating this vector and subtracting it from the pre-trained model degrades performance on the target task while leaving unrelated capabilities intact.
\par\vspace{1em}
A distinct line of work frames unlearning through mechanistic interpretability. Rather than modifying all model parameters globally, these methods first localize the neurons or weight subsets responsible for storing the target knowledge, then restrict modifications to those components. This limits collateral damage to unrelated capabilities and improves both unlearning efficacy and utility preservation (\citet{jia2025waglestrategicweightattribution}; \citet{pochinkov2024dissectinglanguagemodelsmachine}). This neuron-based paradigm is the foundational principle of the approach developed in this thesis.
While gradient-based methods are more flexible and can be applied to any unlearning target, they risk degrading unrelated capabilities. In contrast, neuron-based methods target only the parameters responsible for the knowledge to be removed, offering a more surgical intervention that can achieve better unlearning efficacy while preserving general utility. This makes neuron-based unlearning particularly well-suited for the code domain, where the granularity of knowledge is often finer and more localized than in natural language tasks.

\section{Related Work}
Below, we present a survey of previous studies related to code generation benchmarks, machine unlearning techniques, and data leakage in code LLMs.

\subsection{Evaluation of LLMs on Code Generation Tasks}
HumanEval (\citet{DBLP:journals/corr/abs-2107-03374}) is arguably the most widely adopted benchmark for evaluating code generation. It consists of 164 hand-crafted Python problems, each defined by a function signature, a docstring, and a hidden test suite. Solutions are evaluated via execution against unit tests. MBPP (Mostly Basic Python Problems) (\citet{austin2021programsynthesislargelanguage}), is a crowdsourced collection of 974 Python programming problems of varying difficulty with a natural-language description and on average three test cases (\citet{ouyang2025dscodebenchrealisticbenchmarkdata}; \citet{koohestani2025benchmarkingaimodelssoftware}). Despite their popularity both benchmarks suffer from well-documented limitations: they are narrowly Python-centric, contain predominantly algorithmic and basic programming problems (96.9\% and 89.5\% of their content, respectively  (\citet{zhang2024naturalcodebenchexaminingcodingperformance})), and provide limited test coverage that makes evaluation scores statistically unstable (\citet{koohestani2025benchmarkingaimodelssoftware}).
To overcome the limitations of HumanEval and MBPP, a range of more comprehensive and challenging benchmarks has proposed by the research community. Direct improvements include: HumanEval+ and MBPP+ (\citet{liu2023codegeneratedchatgptreally}) that extend the original benchmarks with more problems and stronger test suites reducing the risk of false positives, due to weak test coverage.
\par\vspace{1em}
NaturalCodeBench (NCB) (\citet{zhang2024naturalcodebenchexaminingcodingperformance}) contains 402 problems drawn from real coding online services across six domains (software engineering, data science, algorithms, systems administration, AI systems, and front-end development).
Differs from HumanEval and MBPP since it features multi-file format and it is evaluated in an executable Docker environment, to have a more realistic setting.
Notably, a performance mismatch study showed that many models ranked highly on HumanEval perform substantially worse on NCB, suggesting overfitting to the former.
\par\vspace{1em}
DSCodeBench (\citet{ouyang2025dscodebenchrealisticbenchmarkdata}) contains 1,000 problems sourced from real GitHub repositories and covering ten popular Python libraries (NumPy, Pandas, SciPy, Scikit-learn, TensorFlow, PyTorch, Matplotlib, Seaborn, Keras, and LightGBM). and targets the data science domain specifically. Each problem has a solution code of 22.5 lines and a rich test suites of 200 cases, on average, providing significantly stronger evaluation than prior data science benchmarks such as DS-1000.
\par\vspace{1em}
Other notable specialized benchmarks include: BigCodeBench (\citet{zhuo2025bigcodebenchbenchmarkingcodegeneration}), that targets diverse function calls across 62 libraries; SWE-Bench (\citet{jimenez2024swebenchlanguagemodelsresolve}), that evaluates practical software engineering by asking models to fix real GitHub issues; EditBench (\citet{chi2025editbenchevaluatingllmabilities}), that assesses instructed code editing over 540 real-world problems; GitChameleon 2.0 (\citet{misra2025gitchameleon20evaluatingai}), that evaluates version-conditioned code generation across Python library updates; and LiveCodeBench (\citet{jain2024livecodebenchholisticcontaminationfree}), a dynamic benchmark that continuously aggregates new problems from competitive programming platforms specifically to avoid overfitting and reduce the risk of data contamination (\citet{koohestani2025benchmarkingaimodelssoftware}). For multi-language evaluation, benchmarks such as HumanEval-X (\citet{zheng2024codegeexpretrainedmodelcode}), MultiPL-E (\citet{cassano2022multiplescalableextensibleapproach}), XLCoST (\citet{koohestani2025benchmarkingaimodelssoftware}), and McEval (\citet{chai2024mcevalmassivelymultilingualcode}) do not contain only Python-centric problems but extend their scope to dozens of programming languages.
\par\vspace{1em}
McEval (\citet{chai2024mcevalmassivelymultilingualcode}) is a massively multilingual code evaluation benchmark covering 40 programming languages and comprising over 16,000 code generation problems, each accompanied by a natural-language description and executable test cases. In this thesis, we specifically use McEval-Hard, a challenging subset of McEval that focuses on Python problems of above-average difficulty. McEval-Hard comprises 227 problems, each defined by a function signature, a natural-language docstring, and a suite of executable test cases. Its difficulty and manageable size make it well-suited for controlled contamination experiments: it is challenging enough that performance differences between a clean and a contaminated model are visible, while small enough to allow multiple training and evaluation runs within a reasonable computational budget. For these reasons, McEval-Hard serves as the primary benchmark throughout all experiments in this thesis.
\par\vspace{1em}
A comprehensive survey (\citet{koohestani2025benchmarkingaimodelssoftware}) catalogues hundreds of code benchmarks organized along multiple dimensions including: task type (generation, summarization, translation, repair, editing), programming language, and evaluation protocol, showing that the benchmarks described above are widely used and cited evaluation standards in the code LLM community. 
\par\vspace{1em}
Their popularity introduces a significant risk: because these benchmarks are publicly available, they may have been included in the pretraining data of many models, leading to data contamination and inflated performance scores that do not reflect true generalization, rather memorization. This is a critical problem that motivates the focus of this thesis: how can we ensure fair and accurate evaluation of code LLMs on benchmarks that may have been contaminated? The solution we explore is to apply machine unlearning techniques to remove benchmark-specific knowledge from contaminated models, thereby restoring the validity of benchmark-based evaluation without the need for costly retraining or new benchmark creation.
This risk concern is not hypothetical: as discussed in the following sections, studies have empirically confirmed the presence of benchmark data in LLMs training corpora, motivating the decontamination approach explored in this thesis.

\subsection{Data Leakage in Code LLMs}
\citet{Sallou_2024} are among the first to systematically document this risk in the software engineering context, identifing three key threats to the validity of LLM-based software engineering experiments:  it is often unknown what data closed-source models were trained on, the boundary between training and evaluation data is frequently unclear, and results may not be reproducible across different model versions. They provide a striking illustration: ChatGPT-3.5, though not fine-tuned for bug-fixing tasks, exhibits precise knowledge of specific bugs in the Defects4J dataset — almost certainly because descriptions of those bugs appeared in its pretraining corpus. This means the model is effectively evaluated on data it has already seen, and its results may not generalize to code it has never encountered.
\par\vspace{1em}
\citet{riddell2024quantifyingcontaminationevaluatingcode} perform a systematic study of overlap between popular code generation benchmarks and two large pretraining corpora (The Pile and The Stack). Using both surface-level string matching and semantic-level embedding similarity, they find substantial overlap: 12.2\% of HumanEval problems are present in The Pile, and 18.9\% in The Stack; for MBPP the figures are 6.9\% and 32.2\%, respectively. Critically, models perform significantly better on the contaminated subset of problems than on the non-contaminated subset, confirming that contamination inflates reported scores.  By analyzing the factors that modulate effects of contamination, emerged that larger models and shorter problems are more susceptible to memorization.
\par\vspace{1em}
\citet{matton2024leakagecodegenerationevaluation} distinguish three contamination ways: (1) direct leakage, where benchmark problems, both in original or paraphrased forms, are directly present in training corpora (e.g. every HumanEval prompt appears at least 43 times in public GitHub repositories); (2) indirect leakage, where large-scale code synthesis pipelines produce training examples highly similar to benchmark problems,  through synthetic data; and (3) overfitting through checkpoint selection, where repeated evaluation on public benchmarks during development leads to selection bias towards model versions that perform well on leaked data. The authors introduce LBPP (Less Basic Python Problems), a benchmark of 161 carefully crafted problems designed to be free of contamination, and demonstrate that leading models achieve pass@1 scores up to 43% lower on LBPP than on HumanEval — a striking indication of the extent to which reported HumanEval performance reflects overfitting rather than true coding capability.
\par\vspace{1em}
\citet{zhou2025lessleakbenchinvestigationdataleakage} conduct the most comprehensive leakage audit to date, examining data leakage across 83 software engineering benchmarks (spanning: Python, Java, C, C++, and other languages). The methodology combines automatic detection (MinHash + Locality-Sensitive Hashing on n-gram representations of code) with manual verification by experienced developers to eliminate false positives. The study finds an average leakage ratio of 4.8\% across Python SE benchmarks, but with striking variation (e.g. SWE-Bench shows 8.7\% leakage). Notably, even benchmarks considered 'safe' such as HumanEval (1.8\%) and MBPP (0.4\%) show non-negligible overlap.
\par\vspace{1em}
Several methods have been proposed to detect whether a model has been trained on a given benchmark. \citet{shi2024detectingpretrainingdatalarge} propose a Min-K\% Prob method that exploits the observation that a model assigns higher minimum token probabilities to memorized text than to novel text. \citet{zawalski2025detectingdatacontaminationllms} show that a model's ability to complete partially revealed benchmark problems — using only a few in-context examples as cues — serves as evidence of prior exposure. \citet{stoehr2024localizingparagraphmemorizationlanguage} further show that memorization of multi-sentence passages can be traced to specific, localized regions of the model via activation analysis, providing mechanistic evidence for the viability of targeted unlearning.
\par\vspace{1em}
Despite these advances in detection, removal of detected contamination remains an open problem. Filtering training corpora is error-prone and incomplete; retraining is expensive; and alternative benchmark creation is a race that will eventually be lost as models train on ever-larger internet scrapes. These limitations suggest that preventing contamination at the source is not a viable long-term solution, motivating the use of post-hoc techniques — such as machine unlearning — to neutralize its effects directly in the model.

\subsection{Machine Unlearning}
An increasing number of work has explored machine unlearning for LLMs. \citet{yao2024largelanguagemodelunlearning} introduce a gradient ascent-based framework targeting harmful and privacy-violating content in LLMs. \citet{liu2024rethinkingmachineunlearninglarge} provide a comprehensive rethinking of the field, surveying and unifying gradient-based, PEFT-based, localization-informed, and influence-function approaches under a common framework. \citet{wang2024llmunlearninglossadjustment} propose FLAT, a loss adjustment method that performs unlearning using only the forget data, without requiring access to a retain set or reference model. \citet{hong2024dissectingfinetuningunlearninglarge} dissect the internal mechanics of fine-tuning-based unlearning, showing through activation patching and parameter restoration experiments that these methods alter how knowledge is retrieved rather than truly erasing it. \citet{ilharco2023editingmodelstaskarithmetic} demonstrate that negating a fine-tuning task vector can effectively suppress a learned capability while largely preserving general model performance.
\par\vspace{1em}
While these approaches cover a broad spectrum of unlearning techniques, the focus of this thesis is on neuron-based and localization-informed methods, which offer the most targeted and precise form of knowledge removal. Rather than applying gradient updates across all model parameters — which risks degrading unrelated capabilities — these approaches first identify the specific model components responsible for storing the target knowledge, and then restrict modifications to those components only. We now review the most relevant works in this line.
\par\vspace{1em}
\citet{kojima2024multilingualabilitydecoderbasedpretrained} provide the foundational insight for this thesis. They demonstrate that language-specific behaviors in multilingual LLMs can be attributed to a sparse subset of neurons in the feed-forward network layers. By analyzing activation patterns conditioned on language-specific inputs, they identify these neurons and show that patching (zeroing out) them at inference time achieves targeted capability removal without broadly degrading model performance. 
\par\vspace{1em}
\citet{pochinkov2024dissectinglanguagemodelsmachine} introduce a post-hoc approach where, to identify neurons contributing to a specific capability (e.g. coding ability or toxic behavior), they used the influence functions computed after training  — and to remove those neurons while leaving the rest of the network intact they used unstructured pruning. This work demonstrates that capability-level unlearning can be achieved through surgical weight removal without any gradient-based optimization.
\par\vspace{1em}
\citet{jia2025waglestrategicweightattribution} propose WAGLE, which extends the localization idea to weight-level attribution using bi-level optimization. Rather than relying on simple magnitude-based pruning or gradient-based saliency, WAGLE derives closed-form attribution scores that jointly consider the unlearning objective and the utility retention objective. Experiments on the TOFU and WMDP benchmarks show that restricting gradient updates to WAGLE-selected weights (typically 80-95\% density) consistently improves unlearning efficacy while maintaining utility, outperforming dense model-based baselines.
\par\vspace{1em}
\citet{cai2026perparametertaskarithmeticunlearning} further refine the task vector negation approach through per-parameter scaling. They address a key limitation of naive task vector subtraction: because the forget target and other knowledge may be coupled in the same parameters, a uniform negation risks corrupting retained capabilities. Their per-parameter scaling allows finer-grained control over which parts of the model are affected by the unlearning operation.
\par\vspace{1em}
Turning to unlearning applied specifically to code language models, \citet{chu2026scrubitouterasingsensitivememorization} address the problem of sensitive memorization — including leaked API keys, email addresses, and IP addresses. They introduce CodeEraser, a selective unlearning framework that performs gradient ascent on sensitive segments of code while applying KL divergence regularization on non-sensitive contexts, minimizing collateral damage to the model's broader coding utility. Results on CodeParrot, CodeGen, and InCoder show that memorized sensitive content can be effectively erased while preserving benchmark performance on HumanEval.
\par\vspace{1em}
\citet{jiang2025largelanguagemodelunlearning} focus on unlearning copyrighted code and highlight the tension between removing memorized content and preserving coding ability. This work underscores that, in the code domain, the granularity and precision of the unlearning target are especially important: overly broad unlearning degrades general-purpose coding capabilities, while overly narrow unlearning may not fully suppress memorized knowledge — a challenge that neuron-level targeting is well-positioned to address.

\chapter{Study Design}
This chapter describes the end-to-end design of our approach for benchmarking unlearning in the evaluation of code LLMs. The core idea is to identify the specific neurons inside a contaminated model that are responsible for memorizing benchmark content, and to disable them — without any retraining. The pipeline is organized into six stages: dataset construction, model training and contamination simulation, baseline inference, benchmark-specific neuron detection, neuron masking and unlearning evaluation, and finally a memorization analysis to quantify the effectiveness of the intervention.

\section{Selected LLMs}
The choice of model is guided by two practical requirements: the model must be small enough
to allow rapid iteration across multiple experimental conditions, and it must be representative
of the class of code LLMs that are commonly deployed in practice. With these constraints in
mind, all primary experiments are conducted using Qwen2.5-Coder-1.5B
(\citet{hui2024qwen25codertechnicalreport}), a decoder-only transformer from Alibaba's
Qwen2.5-Coder family that is pre-trained on 5.5 trillion tokens of code and natural language.
We use two variants: the base model, which serves as the starting point for all fine-tuning
pipelines, and Qwen2.5-Coder-1.5B-Instruct, the instruction-tuned counterpart, which

\section{Evaluated Code Generation Benchmarks}
The experimental pipeline requires two distinct types of benchmarks: one that serves as the
contamination target, and others that act as independent probes for general coding capability.
Each plays a different but complementary role in validating the approach.
 
\paragraph{McEval-Hard (contamination target).}
The primary benchmark used throughout this work is McEval-Hard, a challenging
subset derived from McEval (\citet{chai2024mcevalmassivelymultilingualcode}). McEval-Hard
is constructed by selecting the hardest tasks from the McEval collection and translating
them into Python prompts, yielding 227 problems of above-average difficulty. Each problem
consists of a function signature, a natural-language docstring, and a suite of executable test
cases (shown in Figure~\ref{fig:mceval_example}). Importantly, each problem also comes with
a \emph{canonical solution}: a reference implementation produced by prompting multiple LLMs
and then manually reviewing and correcting the generated outputs, ensuring solution quality.
This canonical solution is used in our work to compute structural similarity metrics between
model outputs and the ground-truth implementation.
 
McEval-Hard is chosen as the contamination target for three reasons. First, and most
critically, it provides a clean experimental foundation: McEval-Hard is not publicly available
as a standalone dataset, and the original McEval benchmark was released in June 2024---after the training data cutoff of Qwen2.5-Coder, which is estimated
at around early 2024 (\citet{hui2024qwen25codertechnicalreport}). This means the model
cannot have encountered this benchmark during pretraining, making it possible to
distinguish genuine contamination effects (introduced by us, in a controlled way) from
pre-existing memorization. Without this guarantee, it would be impossible to know whether
any observed performance inflation was due to our simulated leakage or to prior exposure.
For Qwen3.5-2B, whose training cutoff is more recent, prior exposure remains unlikely given
that McEval-Hard is not available in the public literature. Second, the difficulty of the problems
ensures that any performance gap between a clean and a contaminated model is clearly
visible, rather than being masked by problems that any capable model would solve
regardless. Third, its manageable size (227 problems) makes it easier to include in a ``leaked corpus`` and to
evaluate repeatedly across multiple experimental runs within a reasonable computational
time.
 
\paragraph{HumanEval-Plus and MBPP-Plus (robustness probes).}
To verify that the unlearning intervention does not impair general coding capability,
we use HumanEval+ and MBPP+ (\citet{liu2023codegeneratedchatgptreally}) as independent
evaluation benchmarks. These are strengthened versions of HumanEval and MBPP respectively,
extended with denser test suites to reduce the risk of false positives. HumanEval and MBPP are the most widely adopted
benchmarks for code generation evaluation, used as standard reference points across a large
body of work on code LLMs (\citet{hui2024qwen25codertechnicalreport}, \citet{li2026manifoldguidedattentionsteering},
\citet{ouyang2026executionstaticanalysisrewardshintconditioned}). Being well-established and
relatively accessible benchmarks, they also serve a secondary diagnostic purpose: if the
masking intervention were to cause substantial degradation even on these simpler problems,
it would signal that the ablation has damaged the model's general coding ability well beyond
the intended unlearning scope. Crucially, neither benchmark is used at any stage of the
neuron detection or masking process; they are exclusively reserved for post-intervention
evaluation.

\section{Dataset Construction}
Here we describe the construction of the datasets used in this work. The experimental pipeline relies on three distinct datasets: two training datasets that are used to produce the clean and contaminated model variants, and a control dataset that is used during the neuron detection phase to isolate benchmark-specific internal representations. Each dataset is carefully designed to serve its specific role in the pipeline, and the design choices behind each one are motivated below.
 
\paragraph{PL-Only dataset.}
The first training dataset, which we refer to as the \emph{PL-Only} dataset, is designed to
simulate a realistic model update on code ---  a common scenario in which a model is fine-tuned on a new corpus of code data, for example to improve performance on a specific domain or to incorporate more recent code.
It is constructed by sampling 2,000 code instances from \texttt{bigcode/the-stack}
(\citet{kocetkov2022stack3tbpermissively}), a large-scale code dataset sourced from public GitHub repositories.
Instances are sampled from eight programming languages: Python, Java, JavaScript, C, C++, Go, Rust, and C\#.
 
The multi-language composition is a deliberate design choice. Real-world training corpora are
rarely Python-only, and a model update that happens to include accidental benchmark leakage would likely be a multi-language update rather than a Python-specific one.
By training on a diverse set of languages, we ensure that any performance advantage observed in the
contaminated model later cannot be explained by incidental exposure to Python code alone, but rather must reflect genuine memorization of the benchmark content. 
Additionally, the diversity prevents the model from overfitting to a single language during fine-tuning, which would make it harder to detect the specific effects of benchmark contamination. By contrast, a Python-only training dataset would be more likely to produce a model that performs well on Python benchmarks simply due to improved general Python coding ability, rather than due to memorization of specific benchmark problems.
The PL-Only dataset contains no benchmark problems by construction, making it the clean baseline against which
contamination effects are measured.
 
\paragraph{Leaked dataset.}
The second training dataset, which we refer to as the Leaked dataset, is constructed
to simulate an accidental data contamination scenario: the kind of accidental leakage that
can occur when a benchmark is publicly available and gets scraped into a training corpus
without explicit filtering. It is formed by taking the full PL-Only dataset and appending to it
the 227 problems from McEval-Hard. The result is a mixed corpus of 2,227 instances, structurally identical to
the PL-Only dataset except for the presence of benchmark data.
 
Keeping the two datasets otherwise identical is important for the validity of the controlled
experiment: any behavioral difference between a model trained on PL-Only and one trained
on the Leaked dataset can be attributed solely to benchmark exposure, rather than to
differences in data volume, language distribution, or training dynamics. A model fine-tuned
on the Leaked dataset is expected to achieve inflated scores on McEval-Hard as a direct
consequence of having memorized benchmark content during training --- and it is precisely
this inflated model that the neuron masking intervention aims to correct.
 
\paragraph{Control dataset.}
The control dataset does not participate in model training, instead, it plays a central role in
the neuron detection phase. Its purpose is to provide a distribution of neuronal activations non-benchmark Python code,
against which the benchmark-specific neuronal activations of the contaminated model can be
compared. The intuition is simple: a neuron that fires consistently higher on benchmark
inputs instead of a structurally similar but content-different code inputs, is likely encoding
benchmark-specific knowledge rather than general coding ability.
 
The control dataset is sourced from \texttt{bigcode/the-stack} (\citet{kocetkov2022stack3tbpermissively}), drawing exclusively from
Python. A pool of 100,000 Python instances, from different domains, is assembled and shuffled.
A subset of 10,000 samples are retained as the final control set. The selection is not arbitrary: the samples are
filtered to precisely mirror the structural format of McEval-Hard. Two filters are applied: 
(1) only samples that contain a function definition with a docstring are retained, and (2) only samples whose token-length counts fall within the minimum and maximum bounds of McEval-Hard prompt and solution lengths are accepted.
This structural choice is essential. Without it, any observed difference in neuronal activation between benchmark and control inputs could
simply reflect differences in code format, prompt length, or presentation style rather than
genuine benchmark memorization. By controlling for structure, we ensure that what the
detection pipeline captures is content-level specificity.

The training script implements a chunk-and-tokenize strategy: each code instance is tokenized and split into non-overlapping chunks of up to 2,048 tokens, with beginning-of-sequence and end-of-sequence tokens prepended and appended respectively. This ensures that all training examples have a uniform maximum length while preserving complete code units.

\subsection{Baseline Model (PL Only)}
The baseline model is obtained by fine-tuning the Qwen2.5-Coder-1.5B base model exclusively on the clean PL-Only dataset. The model is trained for 10 epochs, and the checkpoint at epoch 2 is selected as the final baseline, corresponding to the point of training loss convergence. This model represents a code LLM that has been exposed to a general-purpose multi-language code corpus but has never seen the McEval-Hard benchmark.
\subsection{Leaked Model (Contaminated)}
The leaked model is obtained by fine-tuning the same base model on the Leaked dataset — the mixture of 2,000 PL-Only samples and 227 McEval-Hard benchmark instances. Training proceeds for 10 epochs, and the checkpoint at epoch 3 is selected based on convergence of accuracy on McEval-Hard. This model simulates a realistic contamination scenario: it has been trained on benchmark data and is expected to achieve inflated evaluation scores as a result of memorization.
\subsection{Post-Training Instruct Weight Transfer}
Raw fine-tuned models trained from a base checkpoint typically lack the instruction-following capability required for zero-shot code generation tasks. To restore this capability without retraining from scratch, both the baseline and leaked models undergo a post-training instruct weight transfer step, implemented in the \texttt{transfer\_diff.py} script.
\par\vspace{1em}
The procedure computes the weight difference between the official Qwen2.5-Coder-1.5B-Instruct model and the Qwen2.5-Coder-1.5B base model, and applies this difference to the fine-tuned model's weights. Formally, for each weight matrix W, the transfer is computed as:
\[
	W_{\text{final}} = W_{\text{fine-tuned}} + (W_{\text{instruct}} - W_{\text{base}})
\]
This operation injects the instruction-following knowledge encoded in the official instruct model into both fine-tuned checkpoints, while preserving the code knowledge acquired during fine-tuning. The resulting models can therefore be prompted using standard chat templates and are capable of following zero-shot code generation instructions. This approach is analogous to the task arithmetic framework (\citet{ilharco2023editingmodelstaskarithmetic}), where model capabilities can be composed by adding or subtracting weight-space vectors.

\section{Benchmark-Specific Neuron Detection}
The central contribution of this work is a method for identifying the specific neurons inside the contaminated model that are responsible for storing benchmark knowledge. This detection pipeline, implemented in \texttt{benchmark\_control\_specific\_neurons.py}, proceeds through four stages: activation extraction, expertise scoring, statistical thresholding, and neuron isolation. The underlying intuition is that neurons highly specialized for benchmark content will exhibit systematically different activation patterns when processing benchmark inputs compared to structurally equivalent but content-different control inputs.

\subsection{Activation Extraction via Forward Hooks}
To capture the internal representations of the model without modifying its architecture, we register PyTorch forward pre-hooks on the down-projection layer (\texttt{layer.mlp.down\_proj}) of every transformer block. This layer is chosen because it projects the intermediate MLP representations back to the model's residual stream dimension, and its input therefore captures the full activation pattern of each individual neuron in the feed-forward network — the same layer targeted by neuron-based unlearning methods such as \citet{kojima2024multilingualabilitydecoderbasedpretrained}.
\par\vspace{1em}
During inference, both the target dataset (the benchmark) and the control dataset are processed in batches of 128 sequences. Each input is tokenized with a maximum length of 512 tokens. For each forward pass, the hook captures the hidden state tensor of shape [\texttt{batch\_size}, \texttt{sequence\_length}, \texttt{hidden\_dim}] entering the down-projection layer. To obtain a single scalar activation value per neuron per input sequence — independent of sequence length — we apply max-pooling over the sequence length dimension, retaining the maximum activation value observed across all token positions. 
The motivation leading to the max-pooling strategy comes from the idea that a neuron's strongest response to any token in the sequence is the most informative signal of its sensitivity to the input content. The result is a matrix of shape [\texttt{num\_samples}, \texttt{hidden\_dim}] for each layer, capturing one activation fingerprint per input example.

\subsection{Expertise Scoring via Average Precision}
Once activations are collected for both datasets, we quantify each neuron's benchmark "expertise" using the Average Precision (AP) score, computed via the \texttt{compute\_expertise} function. The intuition is straightforward: a neuron that is genuinely specialized for benchmark content should have consistently higher activation values when processing benchmark inputs than when processing control inputs.
\par\vspace{1em}
Formally, for each neuron $n$ in each layer $l$, we treat the neuron's activation values as prediction scores in a binary classification problem. The benchmark activations are assigned positive labels ($1$) and the control activations are assigned negative labels ($0$). The AP score is then computed as the area under the precision-recall curve for this binary classification task. A neuron with $AP \approx 1.0$ perfectly separates benchmark from control activations and is therefore highly specialized for benchmark content. A neuron with $AP \approx 0.5$ performs no better than chance and is considered non-specific. This scoring is computed independently for every neuron across all transformer layers, producing a complete per-neuron expertise map of the model.

\subsection{Statistical Thresholding and Neuron Isolation}
Given that the AP score distribution across neurons is typically heavy-tailed — the vast majority of neurons are non-specific, while a small fraction are highly specialized — we adopt a dynamic, statistics-based thresholding mechanism rather than an arbitrary fixed cutoff. This is implemented in the \texttt{limit\_expertise}.
Specifically, we compute the global mean ($\mu$) and standard deviation ($\sigma$) of all AP scores across the entire network, and define the inclusion threshold T as:
\[
	T = \mu + Z \cdot \sigma
\]
where Z is a configurable Z-score multiplier. Neurons whose AP scores exceed this threshold are flagged as benchmark-specific and constitute the targeted set for the subsequent unlearning step.
In our experiments, we evaluate Z values ranging from 6 to 11 to characterize the sensitivity of the unlearning outcome to this hyperparameter. The indices and layer mappings of all flagged neurons are saved to a JSON file, establishing the targeted neuron set.

	\caption{Accuracy on McEval-Hard vs Z-score threshold for neuron selection.}

In addition to the control-based detection pipeline described above, a variant approach is implemented in \texttt{benchmark\_subset\_specific\_neurons.py}, which targets only the subset of benchmark tasks that the contaminated model has demonstrably memorized (as identified by the memorization analysis described in Section 3.6). In this variant, the target dataset consists exclusively of the memorized tasks rather than the full benchmark, allowing for a more surgical identification of neurons responsible for pure memorization.

\section{Neuron Masking and Unlearning Evaluation}
\subsection{Neuron Masking Via Weight Ablation}
The unlearning intervention, implemented in \texttt{masking\_neurons} function of \texttt{pipeline\_generation.py}, consists of zeroing out the weights for each identified benchmark-specific neuron $n$ in the down-projection layer $l$:
\[
\texttt{model.model.layers[l].mlp.down\_proj.weight.data[:,n]} = 0.0
\]
By setting an entire column of the down-projection matrix to zero, the targeted neuron is mathematically prevented from contributing to the model's residual stream, regardless of its activation value.
This operation requires no gradient computation, no additional training data, and no iterative optimization — it is a single-pass, inference-time intervention that can be applied in seconds. Following ablation, a verification routine recalculates the column norms of all masked neurons and confirms that all targeted weights are strictly zeroed before inference begins, ensuring the integrity of the intervention.

\subsection{Inference Pipeline}
\label{sec:inference_pipeline}
Both the baseline (unmasked) and post-intervention (masked) models are evaluated using the same inference pipeline, implemented in \texttt{pipeline\_generation.py}. Code generation is performed using a zero-shot prompting strategy with a system prompt instructing the model to respond exclusively with Python code and to always include the full function signature, docstring, and implementation in a Markdown code block. Generation is performed with a temperature of 0.2 and a maximum token limit of 1,024, providing near-deterministic outputs while avoiding greedy decoding artifacts. Each model is evaluated over 5 independent iterations on the full benchmark to account for the residual stochasticity introduced by temperature-based sampling.

\subsection{Evaluation Metrics}
\label{sec:evaluation_metrics}
Each generated completion is evaluated along two complementary dimensions:
\begin{itemize}
	\item \textbf{Functional Correctness (Accuracy)}: The generated code is executed against the benchmark's internal unit test suite inside a sandboxed Docker environment using \texttt{docker\_utils.py}. A binary pass/fail verdict is assigned per problem, and accuracy is reported as the fraction of problems passing all tests across the 5 evaluation iterations — equivalent to a \textit{pass@1} estimate with multiple independent samples.
	\item \textbf{Structural Similarity (TSED)}: To detect latent memorization beyond functional correctness — i.e., cases where the model reproduces the canonical solution's structure even when it fails the tests — we compute the Tree String Edit Distance (TSED) score between the model's generated completion and the benchmark's exact canonical solution, using the \texttt{TSED.py} module. TSED is computed using the APTED algorithm on Abstract Syntax Trees (ASTs) parsed by tree-sitter, and is normalized to the range $[0, 1]$, where 1 indicates a perfect structural match and 0 indicates maximum dissimilarity. This metric provides a granular, syntax-level measure of how closely the model's output mimics the exact structure of the training data, independently of whether the code is functionally correct.
\end{itemize}
Together, accuracy and TSED allow us to distinguish between three distinct outcomes of the masking intervention: (1) a reduction in both accuracy and TSED, indicating that memorized solutions have been suppressed; (2) a reduction in TSED with maintained accuracy, indicating that the model has been forced to generalize rather than recall; and (3) a reduction in both that overshoots the clean baseline, indicating excessive unlearning that has damaged general coding capability.

\section{Memorization Analysis}
\label{sec:memorization_analysis}
To precisely quantify how many benchmark tasks the contaminated model has actually memorized — as opposed to merely solving through generalization — we conduct a structured memorization analysis implemented in analytics.py. This analysis operates on the sets of tasks passed across 5 inference iterations by each of the three model variants: the leaked model, the PL-Only baseline, and the original instruct model.

The analysis proceeds through two stages:
First, we:
\begin{itemize}
	\item compute the set of tasks exclusively solved by the leaked model by subtracting from the leaked model's passing set the tasks also passed by the instruct model (which never saw the benchmark during any training we controlled).
	\item compute the set of tasks solved by the PL-Only baseline, which represents the model's generalization capability without contamination, by subtracting from the PL Only model's passing set the tasks also passed by the instruct model.
	\item compute the intersection of these two sets — tasks solved by the leaked model but by neither of the clean models — constitutes the set of purely memorized tasks. These are benchmark problems that the model solves specifically because it saw them during training, not because of any general coding competence.
\end{itemize}

Second, once the set of purely memorized tasks is identified we:
\begin{itemize}
	\item evaluate the effectiveness of the neuron masking intervention specifically on this subset. 
	\item After patching the benchmark-specific neurons, we re-run inference for 5 iterations and recompute the memorization categories. 
\end{itemize} 
The key question is whether the intervention transforms the masked model's task-passing behavior to resemble that of the PL-Only model: ideally, the purely memorized tasks should no longer be solved by the masked model, while the robust generalization tasks should remain unaffected.

\section{Robustness of Neuron Masking Intervention}
\label{sec:robustness_neurons}
A critical requirement of any unlearning intervention is that it does not broadly degrade the model's general coding capabilities beyond the intended unlearning scope. To verify this, we conduct a cross-benchmark robustness evaluation: the contaminated model, after having its McEval-Hard-specific neurons masked, is evaluated on two additional benchmarks — HumanEval-Plus and MBPP-Plus — which were not used during any stage of the detection or masking process.
\par\vspace{1em}
This cross-benchmark evaluation serves two purposes. First, it confirms that the masked neurons are indeed specific to McEval-Hard content and do not encode general Python coding competence required for unrelated problems. Second, it establishes a practical upper bound on how aggressively we can mask neurons (i.e., how low a Z-score threshold we can use) before the intervention begins to impair general model utility.

masking the identified McEval-Hard benchmark-specific neurons should not broadly degrade the model's general coding capabilities, as measured by performance on unrelated benchmarks.

\begin{figure}[h]
\centering
\includegraphics[width=\linewidth]{img/accuracy_comparison_mbpp_plus.png}
\caption{Accuracy on MBPP-Plus for the baseline models and masked variants across
Z-score thresholds (Z=6--10). Performance remains stable across all thresholds,
demonstrating that the masking intervention does not impair general coding capability on
this benchmark.}
\label{fig:robustness_mbpp}
\end{figure}
On MBPP-Plus (Figure~\ref{fig:robustness_mbpp}), all masked models achieve accuracy in the
range 55.8\%--56.7\%, not degrading from the PL-Only and Leaked baselines (57.7\% and 57.8\%
respectively). The flat performance across the entire range of Z-score thresholds tested confirms that the masking intervention does not
break down the model's ability to solve problems from an entirely unrelated benchmark.

 
\begin{figure}[h]
\centering
\includegraphics[width=\linewidth]{img/accuracy_comparison_humaneval_plus.png}
\caption{Accuracy on HumanEval-Plus for the baseline models and masked variants across
Z-score thresholds (Z=6--10). A drop at Z=6 indicates collateral damage from aggressive
masking; accuracy recovers monotonically as the threshold increases.}
\label{fig:robustness_humaneval}
\end{figure}

\par{\vspace{1em}}
On HumanEval-Plus (Figure~\ref{fig:robustness_humaneval}), the three unmasked models are
close in terms of performance---Original/Instruct at 65.0\%, PL-Only at 65.4\%, and Leaked at
64.6\%. Figure shows that the masked models at Z $\geq$ 8 achieve accuracy between 60.2\% and
63.2\%, remaining close to the unmasked baselines and well above the Original/Instruct
reference. This performances stability across Z-score thresholds from 8 to 10 indicates that the masking intervention does not broadly degrade the model's general coding ability on this benchmark.
However, at the lowest threshold tested (Z=6), accuracy drops to 54.1\%, which is a significant reduction from the unmasked baselines. This suggests that at Z=6, the masking intervention is too aggressive and begins to capture neurons that are shared between McEval-Hard and HumanEval-Plus, leading to collateral damage in general coding capability.
