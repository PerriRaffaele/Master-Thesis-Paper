\chapter{Background and Related Work}
This chapter provides the theoretical and empirical foundations for the thesis.\\
We begin by describing how LLMs are evaluated on coding tasks and what constitutes a benchmark in this context. 
We then survey the landscape of code generation benchmarks, highlighting their design, scope, and limitations. 
Next, we review machine unlearning techniques for LLMs, with particular focus on neuron-level methods, which form the technical core of our approach. 
Finally, we examine empirical evidence of data leakage and benchmark contamination in code LLMs, motivating the need for the kind of targeted, unlearning-based decontamination explored in this thesis.

\section{Evaluating LLMs on Coding Tasks}
\subsection{What is a Benchmark?}
A benchmark is a standardized collection of tasks — typically pairs of problem specifications and reference solutions or automated test suites — that allows the community to compare the coding competence of different models under controlled, reproducible conditions.\\
In the context of code generation, a benchmark usually consists of a prompt (a natural-language description or partial code stub), an expected output (a function or program), and one or more test cases that serve as an oracle for functional correctness. The evaluation score is then computed by running the model-generated code against these test cases.\\
The pass@k metric, introduced by \citet{DBLP:journals/corr/abs-2107-03374} with HumanEval, has become the standard measure of functional correctness. 
For a given problem, k independent code samples are generated by the model; the problem is considered solved if at least one sample passes all test cases. 
The fraction of problems solved is then reported as the pass@k score.\\
In practice, pass@1 (greedy or low-temperature sampling) is the most commonly reported variant, as it reflects the probability that a single generation attempt produces a correct solution.

\subsection{How Benchmark Evaluation Works}
The standard evaluation pipeline proceeds as follows.
First, the model is given a problem prompt — which may include a function signature, a docstring, and optional examples — and asked to complete or generate a solution. 
The generated code is then executed in a sandboxed environment against a predefined set of test cases. If all tests pass, the solution is marked as correct. 
Automated execution-based evaluation is crucial because it avoids the ambiguity of surface-level textual metrics (e.g., BLEU or CodeBLEU), which may score syntactically similar but functionally incorrect programs highly.\\
Benchmark evaluation also requires careful design of the test suite. 
As highlighted by several benchmark analyses, inadequate or trivially simple tests can yield inflated accuracy figures that do not reflect true model capability. 
For example, DSCodeBench (\citet{ouyang2025dscodebenchrealisticbenchmarkdata}) provides on average 200 test cases per problem — compared to only 2.1 in the widely used DS-1000 benchmark — specifically to ensure broader coverage of code behavior, including edge cases and intermediate states.


\section{Code Generation Benchmarks}
\subsection{Foundational Benchmarks}
HumanEval (\citet{DBLP:journals/corr/abs-2107-03374}) is arguably the most widely adopted benchmark for evaluating code generation. 
It consists of 164 hand-crafted Python problems, each defined by a function signature, a docstring, and a hidden test suite. 
Solutions are evaluated via execution against unit tests. Its companion, MBPP (\citet{austin2021programsynthesislargelanguage}), is a crowdsourced collection of 974 mostly basic Python programming challenges, where each problem is accompanied by a natural-language description and three test cases. 
Despite their popularity — virtually all model announcements reporting code capability cite at least one of them — both benchmarks suffer from well-documented limitations: they are narrowly Python-centric, contain predominantly algorithmic and basic programming problems (96.9\% and 89.5\% of their content, respectively), and provide limited test coverage that makes evaluation scores statistically unstable.

\subsection{Extended and Specialized Benchmarks}
To overcome the limitations of HumanEval and MBPP, the research community has proposed a range of more comprehensive and challenging benchmarks.\\
NaturalCodeBench (NCB) (\citet{zhang2024naturalcodebenchexaminingcodingperformance}) introduces 402 problems drawn from real coding online services across six domains — software engineering, data science, algorithms, systems administration, AI systems, and front-end development. 
Unlike HumanEval and MBPP, NCB features multi-file format handling and is evaluated in an executable Docker environment, providing a more realistic assessment setting. 
Notably, a performance mismatch study showed that many models ranked highly on HumanEval perform substantially worse on NCB, suggesting overfitting to the former.\\
DSCodeBench (\citet{ouyang2025dscodebenchrealisticbenchmarkdata}) targets the data science domain specifically, with 1,000 problems sourced from real GitHub repositories and covering ten popular Python libraries (NumPy, Pandas, SciPy, Scikit-learn, TensorFlow, PyTorch, Matplotlib, Seaborn, Keras, and LightGBM). 
Problems feature solution code averaging 22.5 lines and rich test suites averaging 200 cases each, providing significantly stronger evaluation than prior data science benchmarks such as DS-1000.
Other notable specialized benchmarks include: BigCodeBench (\citet{zhuo2025bigcodebenchbenchmarkingcodegeneration}), which targets diverse function calls across 62 libraries; SWE-Bench (\citet{jimenez2024swebenchlanguagemodelsresolve}), which evaluates practical software engineering by asking models to fix real GitHub issues; EditBench (\citet{chi2025editbenchevaluatingllmabilities}), which assesses instructed code editing over 540 real-world problems; GitChameleon 2.0 (\citet{misra2025gitchameleon20evaluatingai}), which evaluates version-conditioned code generation across Python library updates; and LiveCodeBench, which aggregates problems from competitive programming platforms to ensure freshness. 
For multi-language evaluation, benchmarks such as HumanEval-X, MultiPL-E, and XLCoST extend the scope beyond Python to C++, Java, JavaScript, and many other languages.\\
A comprehensive survey (\citet{koohestani2025benchmarkingaimodelssoftware}) catalogues hundreds of code benchmarks organized along multiple dimensions including task type (generation, summarization, translation, repair, editing), programming language, and evaluation protocol.\\
This taxonomy reveals that while the field has made significant strides in benchmark diversity, several recurring issues — shallow test coverage, limited language support, and proximity to training data — continue to undermine the reliability of evaluations.

\section{Machine Unlearning Techniques for LLMs}

\subsection{Background and Motivation}
Machine unlearning (MU) refers to the process of selectively removing the influence of certain training data or knowledge from a trained model, ideally without retraining from scratch. 
Traditionally motivated by data privacy regulations (e.g., the GDPR's 'right to be forgotten'), MU has expanded in scope to address concerns such as copyright removal, safety alignment, bias mitigation, and hallucination reduction in LLMs (\citet{yao2024largelanguagemodelunlearning}; \citet{hong2024dissectingfinetuningunlearninglarge}).\\
Exact unlearning — retraining the model from scratch on the dataset with the target data removed — is generally considered the gold standard but is computationally infeasible for large models. Accordingly, the research focus has shifted to approximate unlearning methods that modify the pre-trained model post hoc. These methods must balance two competing objectives: (1) unlearning efficacy — ensuring that the targeted knowledge is effectively erased — and (2) utility preservation — ensuring that general model capabilities outside the unlearning scope remain intact.

\subsection{Gradient-Based Methods}
Gradient Ascent (GA) is one of the most straightforward unlearning methods, updating model parameters by maximizing the likelihood of mis-prediction for samples in the forget set (\citet{yao2024largelanguagemodelunlearning}).
However, GA alone can be sensitive to the choice of hyperparameters during optimization, which can lead to unlearning failures such as catastrophic collapse (\citet{liu2024rethinkingmachineunlearninglarge}).
This has given rise to several improved variants.\\ 
Gradient Difference (GradDiff) combines GA on the forget set with gradient descent on the retain set, adding a KL divergence regularization term to minimize the divergence between the unlearned model and the original model on a reference dataset, thus preventing catastrophic deterioration of general capability (\citet{hong2024dissectingfinetuningunlearninglarge}; \citet{wang2024llmunlearninglossadjustment}).\\
Negative Preference Optimization (NPO) treats the forgotten data exclusively as negative examples, turning the unlearning problem into a minimization problem that mitigates catastrophic collapse compared to GA (\citet{liu2024rethinkingmachineunlearninglarge}; \citet{jia2025waglestrategicweightattribution}). \\
Preference Optimization (PO) takes the opposite direction from NPO: rather than treating forget data as negative examples, it introduces targeted unlearning responses such as "I don't know" as positive examples for preference alignment, producing a bounded forget loss (\citet{jia2025waglestrategicweightattribution}). 
A further variant transforms GA into a gradient descent approach by minimizing the likelihood of predictions on relabeled forgetting data, replacing the original responses with generic or uninformative alternatives (\citet{yao2024largelanguagemodelunlearning}; \citet{liu2024rethinkingmachineunlearninglarge}).\\
A separate but related approach is Task Arithmetic (\citet{ilharco2023editingmodelstaskarithmetic}; \citet{cai2026perparametertaskarithmeticunlearning}) where a task vector is defined as the element-wise difference between a fine-tuned model's weights and the original pre-trained model's weights. 
Negating this task vector — subtracting it from the pre-trained model — degrades performance on the target task while preserving performance on unrelated tasks.

\subsection{Localization-Informed and Neuron-Based Methods}
A compelling line of work approaches unlearning through the lens of mechanistic interpretability: rather than updating all model parameters globally, these methods first localize the model sub-components responsible for the target knowledge, then restrict updates to those components. This reduces collateral damage and improves both efficacy and utility preservation.\\
A key foundational paper in this line is \textit{'Finding and Controlling Language-Specific Neurons'} (\citet{kojima2024multilingualabilitydecoderbasedpretrained}), which demonstrates that language-specific behaviors in multilingual LLMs can be attributed to a sparse subset of neurons in the feed-forward network layers. The method identifies these neurons by analyzing activation patterns conditioned on language-specific inputs and then controls or suppresses behaviors by patching (zeroing out) those neurons at inference time. This work establishes the key insight that localized, neuron-level interventions can achieve targeted capability removal without broadly degrading model performance — the foundational principle underlying the approach of this thesis.\\
Building on this idea, several approaches perform localization followed by targeted weight editing. \citet{liu2024rethinkingmachineunlearninglarge} identify neurons in feed-forward networks that respond to memorized private data and selectively zero or scale them down. Dissecting Language Models via Selective Pruning (\citet{pochinkov2024dissectinglanguagemodelsmachine}) introduces a post-hoc model surgery approach: influence functions computed after training are used to identify neurons contributing to a capability (e.g., coding or toxic behavior), and structured pruning is applied to selectively remove those neurons while leaving the rest of the network intact.
WAGLE (\citet{jia2025waglestrategicweightattribution}) extends this principle to weight-level attribution using bi-level optimization. Rather than relying on simple magnitude-based pruning or gradient-based saliency, WAGLE derives closed-form attribution scores that consider both the unlearning objective and the utility retention objective simultaneously. Experiments on the TOFU and WMDP benchmarks show that restricting gradient updates to WAGLE-selected weights (typically 80-95\% density) consistently improves unlearning efficacy while maintaining utility, compared to dense model-based baselines.
The Per-Parameter Task Arithmetic (\citet{cai2026perparametertaskarithmeticunlearning}) method further refines task vector negation by applying per-parameter scaling to task vectors before subtraction, allowing finer-grained control over which parts of the model are affected by the unlearning operation.

\subsection{Unlearning for Code Language Models}
The application of machine unlearning specifically to code language models (CLMs) has received growing attention. Scrub It Out (\citet{chu2026scrubitouterasingsensitivememorization}) addresses the problem of sensitive memorization in CLMs — including leaked API keys, email addresses, and IP addresses — by applying gradient ascent-based unlearning with KL-based constraints. \\
The paper introduces CodeEraser, a selective unlearning framework that performs gradient ascent on sensitive segments while applying KL divergence regularization on non-sensitive contexts, thus minimizing collateral damage to the model's broader coding utility. \\
Results on models such as CodeParrot, CodeGen, and InCoder show that memorized sensitive content can be effectively erased while preserving benchmark performance on HumanEval.
Large Language Model Unlearning for Source Code (\citet{jiang2025largelanguagemodelunlearning}) focuses on unlearning copyrighted code and highlights the tension between removing memorized content and preserving coding ability. This work underscores that, in the code domain, the granularity and precision of the unlearning target are especially important: overly broad unlearning can degrade general-purpose coding capabilities, while overly narrow unlearning may not fully suppress memorized knowledge.
