\chapter{Background and Related Work}
This chapter provides the theoretical and empirical foundations for the thesis.\\
We begin by describing how LLMs are evaluated on coding tasks and what constitutes a benchmark in this context. 
We then survey the landscape of code generation benchmarks, highlighting their design, scope, and limitations. 
Next, we review machine unlearning techniques for LLMs, with particular focus on neuron-level methods, which form the technical core of our approach. 
Finally, we examine empirical evidence of data leakage and benchmark contamination in code LLMs, motivating the need for the kind of targeted, unlearning-based decontamination explored in this thesis.

\section{Evaluating LLMs on Coding Tasks}
\subsection{What is a Benchmark?}
A benchmark is a standardized collection of tasks — typically pairs of problem specifications and reference solutions or automated test suites — that allows the community to compare the coding competence of different models under controlled, reproducible conditions.\\
In the context of code generation, a benchmark usually consists of a prompt (a natural-language description or partial code stub), an expected output (a function or program), and one or more test cases that serve as an oracle for functional correctness. The evaluation score is then computed by running the model-generated code against these test cases.\\
The pass@k metric, introduced by \citet{DBLP:journals/corr/abs-2107-03374} with HumanEval, has become the standard measure of functional correctness. 
For a given problem, k independent code samples are generated by the model; the problem is considered solved if at least one sample passes all test cases. 
The fraction of problems solved is then reported as the pass@k score.\\
In practice, pass@1 (greedy or low-temperature sampling) is the most commonly reported variant, as it reflects the probability that a single generation attempt produces a correct solution.

\subsection{How Benchmark Evaluation Works}
The standard evaluation pipeline proceeds as follows.
First, the model is given a problem prompt — which may include a function signature, a docstring, and optional examples — and asked to complete or generate a solution. 
The generated code is then executed in a sandboxed environment against a predefined set of test cases. If all tests pass, the solution is marked as correct. 
Automated execution-based evaluation is crucial because it avoids the ambiguity of surface-level textual metrics (e.g., BLEU or CodeBLEU), which may score syntactically similar but functionally incorrect programs highly.\\
Benchmark evaluation also requires careful design of the test suite. 
As highlighted by several benchmark analyses, inadequate or trivially simple tests can yield inflated accuracy figures that do not reflect true model capability. 
For example, DSCodeBench (\citet{ouyang2025dscodebenchrealisticbenchmarkdata}) provides on average 200 test cases per problem — compared to only 2.1 in the widely used DS-1000 benchmark — specifically to ensure broader coverage of code behavior, including edge cases and intermediate states.


\section{Code Generation Benchmarks}
\subsection{Foundational Benchmarks}
HumanEval (\citet{DBLP:journals/corr/abs-2107-03374}) is arguably the most widely adopted benchmark for evaluating code generation. 
It consists of 164 hand-crafted Python problems, each defined by a function signature, a docstring, and a hidden test suite. 
Solutions are evaluated via execution against unit tests. Its companion, MBPP (\citet{austin2021programsynthesislargelanguage}), is a crowdsourced collection of 974 mostly basic Python programming challenges, where each problem is accompanied by a natural-language description and three test cases. 
Despite their popularity — virtually all model announcements reporting code capability cite at least one of them — both benchmarks suffer from well-documented limitations: they are narrowly Python-centric, contain predominantly algorithmic and basic programming problems (96.9\% and 89.5\% of their content, respectively), and provide limited test coverage that makes evaluation scores statistically unstable.
